# 🔐 Security Policy

BRAHM-AI takes the security of its platforms, APIs, users, data, AI services, and infrastructure seriously.

This policy explains how to report a suspected security vulnerability responsibly.

---

## 📦 Covered Systems

This security policy applies to supported BRAHM-AI services, including:

- `https://brahm-ai.in`
- BRAHM True
- BRAHM Voice
- BRAHM-AI Developer APIs
- BRAHM AI Agents
- PotliPay and account services
- Garudaa model and inference services
- Authentication and user-session systems
- Official BRAHM-AI GitHub repositories
- Official backend and integration services

Third-party services, external providers, abandoned experiments, and unofficial copies may not be covered.

---

## 🚨 Reporting a Vulnerability

Do not publish security vulnerabilities in a public GitHub Issue, Discussion, social-media post, video, or public repository.

Send the report privately to:

**Email:** info@ramcoin.org

Use the following subject:

`Security Report — BRAHM-AI`

---

## 📝 Information to Include

Please include as much of the following information as possible:

- Affected website, page, API route, or service
- Type of vulnerability
- Clear reproduction steps
- Expected and actual behaviour
- Potential security impact
- Browser, operating system, or device
- Relevant request and response details
- Screenshots or a non-destructive proof of concept
- Suggested mitigation, when available
- Your preferred contact details

Remove passwords, access tokens, API keys, private user data, payment information, and unrelated personal information before sending the report.

---

## ✅ Responsible Testing

Security testing must remain limited, non-destructive, and respectful of user privacy.

A researcher must not:

- Access or download another user’s data
- Modify or delete production data
- Interrupt platform availability
- Perform denial-of-service or load testing
- Execute automated mass scanning
- Upload malware or destructive payloads
- Attempt credential theft
- Bypass payment, quota, or account restrictions
- Share discovered credentials or private information
- Use social engineering against users or team members
- Continue testing after confirming a vulnerability
- Publicly disclose the issue before coordinated resolution

Testing does not create permission to access restricted systems.

Written authorization is required for penetration testing, infrastructure scanning, model extraction, prompt extraction, API enumeration, or any activity outside normal public product usage.

---

## 🔑 Exposed Credentials

If you discover an exposed:

- API key
- Access token
- Password
- Private key
- Database credential
- Environment variable
- Webhook secret
- Payment credential
- Cloud-service credential

Do not use it.

Report it immediately and include only enough information to identify its location safely.

---

## 🤖 AI and Model Security

Please report issues involving:

- System prompt exposure
- Unauthorized model access
- Model endpoint bypass
- Cross-user memory leakage
- Private conversation exposure
- RAG or knowledge-base data leakage
- Authentication bypass
- Quota or billing bypass
- Unsafe tool execution
- Unauthorized automation actions
- Prompt injection affecting protected systems
- Model or dataset extraction
- API-key exposure
- Cross-account data access

Normal inaccurate AI responses, unsupported questions, or general model hallucinations are not automatically security vulnerabilities unless they create a demonstrable security, privacy, or authorization impact.

---

## 📷 Camera, Voice, and Biometric Data

Reports involving camera, microphone, voice, face-recognition signals, OCR documents, or user memory should be handled carefully.

Do not retain, publish, or share:

- User camera frames
- Voice recordings
- Face images
- Face signatures
- Private documents
- Medicine images containing personal information
- Conversation records
- Location information
- Account identifiers

Include only the minimum evidence required to explain the vulnerability.

---

## ⏱️ Response Process

BRAHM-AI will aim to:

1. Acknowledge a valid report within 5 business days.
2. Review the reported issue and determine its severity.
3. Request additional information when required.
4. Develop and test an appropriate mitigation.
5. Coordinate disclosure after the issue has been resolved.

Response and resolution times may vary depending on complexity, affected services, third-party dependencies, and severity.

Submitting a report does not guarantee that the issue will be classified as a vulnerability.

---

## 📢 Coordinated Disclosure

Do not publicly disclose a vulnerability until:

- BRAHM-AI confirms that the issue has been resolved, or
- A written disclosure timeline has been agreed upon.

Public disclosure without coordination may expose users and systems to unnecessary risk.

---

## 💰 Bug Bounty

BRAHM-AI does not currently operate a guaranteed public bug-bounty program.

Reports are submitted voluntarily unless a written reward agreement has been approved before testing.

Submission of a vulnerability does not create an automatic right to payment, employment, partnership, public credit, or platform access.

---

## 🛡️ Safe and Good-Faith Reporting

BRAHM-AI values responsible security reports made in good faith.

Good-faith reporting means:

- Minimizing access to data
- Avoiding harm
- Stopping after confirming the issue
- Reporting privately
- Following written instructions
- Giving the team reasonable time to investigate and resolve the issue

This policy does not authorize activity that violates applicable law, third-party rights, contracts, or explicit platform restrictions.
